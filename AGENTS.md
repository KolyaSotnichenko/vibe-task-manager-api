# AGENTS.md

Цей документ визначає обовʼязкові правила та найкращі практики для OpenCode-агентів при роботі з Nest.js API у цьому репозиторії.
Мета — забезпечити стабільні API-контракти, передбачувану архітектуру та актуальну документацію.

## Область дії

- Тільки backend (Nest.js API).
- Архітектура, API-контракти, DTO, валідація, документація.
- Не описує бізнес-логіку продукту або інфраструктурні рішення, якщо це явно не запитано в задачі.

## Загальні принципи Nest.js

- **Модульність**: кожен домен — окремий `Module` з чітким public API.
- **Структура модуля**: всередині модуля код розділяється за ролями (`controller`, `service`, `dto`, `entities`, `mappers`, `repository` за потреби). Змішування ролей в одному файлі заборонено.
- **Public vs private API модуля**: у `exports` дозволяється виносити лише стабільні сервіси або порти, які є частиною контракту модуля. Внутрішні реалізації не експортуються.
- **Single Responsibility**: уникай “god‑services”. Якщо сервіс містить декілька незалежних причин для змін — його потрібно розділити.
- **Controllers**: лише транспортний рівень (HTTP ↔ DTO), без бізнес-логіки.
- **Services**: вся бізнес-логіка та взаємодія з іншими сервісами.
- **Dependency Injection**: не створюй інстанси вручну, використовуй DI.
- **Міжмодульна взаємодія**: імпортуй тільки модулі, не напряму сервіси.
- **Глобальні налаштування**:
  - `ValidationPipe` з `whitelist: true`, `forbidNonWhitelisted: true`.
  - Єдиний підхід до exception handling (Nest HTTP exceptions).

## Дизайн API

- REST-орієнтовані ресурси, іменники у множині (`/users`, `/orders`).
- HTTP-методи відповідають дії (GET, POST, PUT/PATCH, DELETE).
- **Версіонування**: не змінюй існуючі контракти без версії.
- **Ідемпотентність**:
  - GET, PUT, DELETE — ідемпотентні.
  - POST — ні, якщо явно не реалізовано інакше.
- **Пагінація**:
  - Явні параметри (`limit`, `offset` або `page`, `perPage`).
  - Стабільний формат відповіді.

## Структура проєкту

- Кожен домен має власну директорію та `Module`.
- Типова структура модуля:
  - `*.controller.ts` — HTTP транспорт.
  - `*.service.ts` — бізнес-логіка.
  - `dto/` — вхідні та вихідні DTO.
  - `entities/` — ORM‑сутності (виключно для persistence шару).
  - `mappers/` — мапінг між entities ↔ DTO.
  - `repositories/` — абстракції доступу до даних (якщо використовується ORM або зовнішнє сховище).
- Крос‑доменно спільний код розміщується в `common/` або `shared/` і не містить бізнес‑логіки конкретного домену.
- Іменування файлів і класів має бути однозначним та відображати роль (`CreateUserDto`, `UsersController`).

## Controllers: транспортний шар

- Controller не містить бізнес‑правил і не працює з ORM напряму.
- Єдиний обовʼязок: прийняти запит → валідований DTO → виклик сервісу → повернути response DTO.
- `try/catch` дозволений лише для мапінгу доменних помилок у HTTP‑винятки.
- HTTP статуси повинні відповідати семантиці операції (201 для create, 204 для delete без body).

## Services: бізнес‑логіка

- Уся доменна логіка зосереджена в services.
- Сервіс визначає транзакційні межі (на рівні use‑case, а не окремих репозиторіїв).
- Заборонено повертати ORM‑entities за межі сервісу.
- Доменні помилки мають бути явними та мапитись на Nest exceptions на межі controller.
- Взаємодія з іншими модулями відбувається лише через їх public API.

## DTO, валідація та трансформація

- **DTO обовʼязкові** для всіх:
  - request body
  - query params
  - response
- Використовуй `class-validator` та `class-transformer`.
- Заборонено:
  - `any`
  - inline-валідація в controller
  - повернення entity напряму
- Чітко розрізняй:
  - `optional` (може бути відсутнє)
  - `nullable` (може бути `null`)
- Вхідні та вихідні DTO **завжди різні типи**.
- Значення за замовчуванням, трансформації та касти виконуються через `class-transformer` (`@Transform`, `@Type`).
- При зміні контрактів використовуй версійовані DTO, не ламаючи існуючі.

## Persistence та інтеграції

- Репозиторій є абстракцією, навіть якщо використовується ORM.
- ORM‑сутності не використовуються в controllers і не повертаються клієнту.
- Клієнти зовнішніх API ізольовані в окремих сервісах або модулях.
- Таймаути, retry‑логіка та error‑handling для інтеграцій є обовʼязковими на концептуальному рівні.

## Помилки, винятки та доменні правила

- Заборонено кидати `Error` або повертати сирі помилки бібліотек.
- Кожна помилка має чіткий HTTP‑відповідник (`400`, `404`, `409`, `422`).
- Повідомлення помилок не повинні розкривати внутрішню реалізацію.

## Помилки та логування

- Використовуй стандартні Nest exceptions (`BadRequestException`, `NotFoundException`, тощо).
- Не витікай внутрішні помилки або stack traces в API response.
- Логи:
  - без персональних даних
  - без секретів
  - достатньо контексту для дебагу
- Логування на рівні controller — для вхідних подій, на рівні service — для доменних переходів.
- Обовʼязкова підтримка correlation / request id (концептуальна вимога).
- Заборонено логувати повні request body, якщо вони можуть містити PII.

## OpenAPI / Swagger

OpenAPI є **джерелом правди** для API-контрактів.

- OpenAPI має оновлюватись у тій самій задачі, що і код.
- Response DTO у коді та схемі Swagger повинні повністю співпадати.
- Пагінація та помилки документуються явно.

### Коли обовʼязково генерувати

- Після будь-якої backend-зміни, що торкається:
  - API-ендпоінтів
  - DTO
  - валідації
  - response-контрактів

### Як генерувати

Виконай **першу доступну** команду з цього списку:

- `pnpm openapi:export`
- `npm run openapi:export`
- `pnpm swagger:generate`
- `npm run swagger:generate`

Якщо **жодної команди не існує**:

- це **blocker**
- зафіксуй помилку в логах
- **зупини виконання задачі**
- не вигадуй тимчасові або локальні скрипти

### Перевірка після генерації

- OpenAPI файл реально оновився (timestamp / diff).
- Нові або змінені ендпоінти присутні в схемі.
- DTO-схеми відповідають актуальній валідації.

### Правила Swagger у Nest.js

- Кожен controller:
  - `@ApiTags`
- Кожен endpoint:
  - `@ApiOperation`
  - мінімум один `@ApiResponse`
- Кожен DTO:
  - `@ApiProperty` для всіх полів
- Заборонено:
  - undocumented endpoints
  - розбіжності між runtime та OpenAPI

### Git

- **Не роби git commit** у межах задачі, якщо це явно не вимагається.
- Згенерований OpenAPI коміть лише тоді, коли це є частиною acceptance criteria.

## Тестування

- Unit-тести для services з замоканими залежностями.
- E2E-тести для критичних API-флоу.
- Swagger **не замінює** тести.
- Unit‑тести перевіряють бізнес‑правила, а не фреймворк.
- E2E‑тести перевіряють HTTP‑контракти та інтеграцію шарів.
- Імена тестів мають описувати очікувану поведінку, а не реалізацію.

## Зміни та еволюція API

- Заборонено ламати існуючі контракти без версіонування.
- Депрекейтед поля або ендпоінти документуються та підтримуються протягом перехідного періоду.
- Видалення API можливе лише після формального депрекейшну.

## Git та робочий процес агента

- Агент зобовʼязаний:
  - прочитати `AGENTS.md` перед початком роботи;
  - проінспектувати існуючу архітектуру;
  - внести мінімально необхідні зміни;
  - оновити OpenAPI, якщо змінюється контракт;
  - перевірити узгодженість DTO, коду та документації.
- Заборонено робити спекулятивні або “про всяк випадок” рефактори.

## Агентам заборонено

- Обходити валідацію.
- Тихо змінювати API-контракти.
- Додавати ендпоінти без Swagger-документації.
- Робити масові рефактори поза межами задачі.

Дотримання цього файлу є обовʼязковим для всіх OpenCode-агентів.
