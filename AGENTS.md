# AGENTS.md

Цей документ визначає обовʼязкові правила та найкращі практики для OpenCode-агентів при роботі з Nest.js API у цьому репозиторії.
Мета — забезпечити стабільні API-контракти, передбачувану архітектуру та актуальну документацію.

## Область дії

- Тільки backend (Nest.js API).
- Архітектура, API-контракти, DTO, валідація, документація.
- Не описує бізнес-логіку продукту або інфраструктурні рішення, якщо це явно не запитано в задачі.

## Загальні принципи Nest.js

- **Модульність**: кожен домен — окремий `Module` з чітким public API.
- **Controllers**: лише транспортний рівень (HTTP ↔ DTO), без бізнес-логіки.
- **Services**: вся бізнес-логіка та взаємодія з іншими сервісами.
- **Dependency Injection**: не створюй інстанси вручну, використовуй DI.
- **Міжмодульна взаємодія**: імпортуй тільки модулі, не напряму сервіси.
- **Глобальні налаштування**:
  - `ValidationPipe` з `whitelist: true`, `forbidNonWhitelisted: true`.
  - Єдиний підхід до exception handling (Nest HTTP exceptions).

## Архітектура та структура проєкту

- Кожен домен — окремий каталог і `Module` (наприклад: `users/`, `orders/`).
- Усередині домену:
  - controller — HTTP шар;
  - service — бізнес-логіка;
  - dto — всі request/response DTO;
  - module — декларація залежностей.
- Public API модуля — тільки через exports у `Module`.
- Спільний код дозволений лише в явно виділених shared/common модулях.
- Заборонено створювати god-modules або ділити домени за технічними, а не бізнес-ознаками.

## Controllers: найкращі практики

- Контролер не містить бізнес-логіки.
- Відповідальність контролера:
  - прийом і валідація DTO;
  - виклик service;
  - повернення response DTO.
- Усі `body`, `query`, `params` описуються DTO, без ручного парсингу.
- Не використовуй `@Req()` або `@Res()` без крайньої необхідності.
- Один endpoint = один чіткий use-case.
- Версіонування реалізується на рівні роутів або controller.

## Services та бізнес-логіка

- Service відповідає за одну бізнес-область.
- Один метод — один use-case.
- Межі транзакцій визначаються на рівні service.
- Взаємодія між доменами — лише через публічні сервіси або фасади.
- Service не залежить від HTTP, Swagger або транспортного шару.

## Робота з даними та persistence

- Контролери не залежать від ORM або конкретної БД.
- Використовуй repository або adapter-підхід.
- Entity з persistence шару ніколи не повертається напряму в API.
- Мапінг persistence model → response DTO є явним і контрольованим.

## DTO, валідація та трансформація

- DTO обовʼязкові для:
  - request body;
  - query params;
  - response.
- Використовуй `class-validator` та `class-transformer`.
- Заборонено:
  - `any`;
  - inline-валідація в controller;
  - повернення entity напряму.
- Чітко розрізняй:
  - optional — може бути відсутнє;
  - nullable — може бути `null`.
- Для складних типів:
  - використовуй `@Type()` для дат, чисел, масивів;
  - enums описуй явно;
  - дефолтні значення задай через трансформацію, не в controller.

## Помилки та логування

- Використовуй стандартні Nest exceptions (`BadRequestException`, `NotFoundException`, тощо).
- Не витікай внутрішні помилки або stack traces в API response.
- Логи:
  - без персональних даних;
  - без секретів;
  - достатньо контексту для дебагу.

## Обробка помилок та edge cases

- Викидай HTTP exceptions якомога ближче до джерела помилки.
- `BadRequestException` — для валідації та некоректних вхідних даних.
- `NotFoundException` — лише коли ресурс реально не існує.
- Не використовуй `throw new Error()` у runtime-коді.
- Повідомлення про помилки мають бути стабільними та без внутрішніх деталей.

## Безпека

- Аутентифікація та авторизація реалізуються через Guards і декоратори.
- Контролер має явно показувати, чи endpoint protected.
- Після валідації перевіряй бізнес-інваріанти.
- Заборонено:
  - хардкодити секрети;
  - логувати токени або credentials.

## OpenAPI / Swagger

OpenAPI є джерелом правди для API-контрактів.

### Коли обовʼязково генерувати

- Після будь-якої backend-зміни, що торкається:
  - API-ендпоінтів;
  - DTO;
  - валідації;
  - response-контрактів.

### Як генерувати

Виконай першу доступну команду:

- `pnpm openapi:export`
- `npm run openapi:export`
- `pnpm swagger:generate`
- `npm run swagger:generate`

Якщо жодної команди не існує:

- це blocker;
- зафіксуй помилку в логах;
- зупини виконання задачі;
- не вигадуй тимчасові або локальні скрипти.

### Перевірка після генерації

- OpenAPI файл реально оновився.
- Нові або змінені ендпоінти присутні в схемі.
- DTO-схеми відповідають актуальній валідації.

### Правила Swagger у Nest.js

- Кожен controller:
  - `@ApiTags`.
- Кожен endpoint:
  - `@ApiOperation`;
  - мінімум один `@ApiResponse`.
- Кожен DTO:
  - `@ApiProperty` для всіх полів.
- Заборонено:
  - undocumented endpoints;
  - розбіжності між runtime та OpenAPI.

### Git

- Не роби git commit у межах задачі, якщо це явно не вимагається.
- Згенерований OpenAPI коміть лише коли це є частиною acceptance criteria.

## Тестування

- Unit-тести для services з замоканими залежностями.
- E2E-тести для критичних API-флоу.
- Swagger не замінює тести.

## Робочий процес агента

- Перед змінами прочитай існуючий код домену повністю.
- Не ламай існуючі API-контракти без версіонування.
- Мінімізуй diff і scope змін.
- Після змін перевір:
  - тести;
  - Swagger/OpenAPI;
  - відсутність неявних контрактних змін.

## Агентам заборонено

- Обходити валідацію.
- Тихо змінювати API-контракти.
- Додавати ендпоінти без Swagger-документації.
- Робити масові рефактори поза межами задачі.

Дотримання цього файлу є обовʼязковим для всіх OpenCode-агентів.
